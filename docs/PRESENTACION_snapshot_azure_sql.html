Soberanía de Datos

Estrategia para retomar el control autónomo de los datos de producción alojados en Azure SQL Database y habilitar proyectos de impacto.
Febrero 2026 | Versión 1.0 | Uso interno
Contexto

El problema actual

Nuestra base de datos de producción vive en Azure SQL Database, controlada por un proveedor externo que no otorga acceso directo.

🔒

Sin acceso autónomo

No podemos consultar, analizar ni explotar los datos por cuenta propia.

⏳

Dependencia total del proveedor

Cada reporte o extracción requiere solicitarlo y esperar respuesta.

🚫

Proyectos bloqueados

Automatización, analytics e IA no pueden alimentarse con datos frescos.

⚠️

Sin respaldo propio

No existe copia independiente fuera de la infraestructura del proveedor.

Visión

Lo que buscamos

Un proceso automatizado que genere copias periódicas de nuestros datos en un servidor local propio, para consultarlos libremente.

📊

Reportes libres

Generar reportes cruzando cualquier catálogo, sin pedir permiso ni esperar.

⚡

Datos frescos

Snapshots cada hora (o la frecuencia que se defina). Siempre actualizados.

🛠

Proyectos de impacto

Alimentar automatización, analytics e IA con acceso directo a datos reales.

Escenarios

Nivel de acceso a negociar

La estrategia depende de lo que el proveedor autorice. Existen 4 escenarios posibles:

A

SQL de solo lectura

Usuario db_datareader + endpoint + firewall. Automatización total, máxima flexibilidad.

Ideal
B

Connection string limitada

Acceso a vistas específicas. Snapshot parcial pero funcional. Requiere mapeo.

Viable
C

Export periódico

Proveedor deposita BACPAC/CSV en Blob Storage. Frescura depende de él.

Aceptable
D

Solo vía API

Extracción endpoint por endpoint. Viable pero alto esfuerzo de desarrollo.

Seguridad

Manejo de credenciales

Principios

  • Cero credenciales en código. Nunca en scripts ni repositorios.
  • Encriptación nativa del SO. Windows DPAPI protege los secretos.
  • Rotación periódica. Calendario acordado con proveedor.
  • Solo lectura. El usuario SQL no tendrá permisos de escritura.

Siguiente paso

Acción inmediata

Solicitar al proveedor:

  • Usuario SQL de solo lectura (db_datareader)
  • Endpoint del servidor Azure SQL
  • Apertura de firewall para nuestra IP

Este es un requerimiento estándar. No implica riesgo para el proveedor: es acceso de solo lectura a datos que son nuestros.

Una vez obtenido el acceso:

1-2 días Instalación y conectividad
2-3 días Primer export + import local
3-5 días Automatización + validación
Semana 2 Primer reporte real desde snapshot
