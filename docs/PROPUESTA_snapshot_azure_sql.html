Introducción

Situación actual

La base de datos de producción de la organización se encuentra alojada en Microsoft Azure SQL Database (plataforma como servicio), bajo la administración y gestión de un proveedor externo de servicios tecnológicos.

Si bien el proveedor entrega los datos cuando se solicitan, no otorga acceso directo a la infraestructura de Azure. Esto significa que la organización no dispone de:

  • Acceso al portal de Azure para administrar la base de datos.
  • Credenciales de conexión directa al servidor SQL.
  • Capacidad autónoma para ejecutar consultas sobre los datos en tiempo real.
  • Un respaldo propio e independiente de la información.

Planteamiento del problema

La ausencia de acceso autónomo a los datos de producción genera las siguientes limitaciones operativas:

  1. Dependencia total del proveedor para cualquier consulta, reporte o extracción de información.
  2. Imposibilidad de generar reportes ad-hoc que crucen información entre múltiples catálogos de la base de datos.
  3. Incapacidad de alimentar proyectos internos de automatización, analítica de datos o inteligencia artificial con datos frescos y completos.
  4. Ausencia de un respaldo independiente, lo que representa un riesgo para la continuidad del negocio en caso de que el proveedor falle o la relación comercial termine.
  5. Tiempos de respuesta inaceptables: cada solicitud de datos al proveedor implica comunicación, espera y procesamiento manual.

Justificación

Los datos generados en la operación del negocio son propiedad de la organización. Retomar el control autónomo sobre estos datos no es solamente una mejora operativa, sino una necesidad estratégica que habilita:

  • Generación de reportes automatizados entre catálogos sin intervención del proveedor.
  • Análisis de datos con herramientas propias (Python, DuckDB, Power BI, Excel).
  • Proyectos de alto impacto que requieran acceso libre, frecuente y programable a la información.
  • Respaldo independiente como estrategia de continuidad de negocio.
  • Reducción de tiempos de obtención de información de días/horas a minutos.

Objetivos

Objetivo general

Establecer un proceso automatizado que genere y descargue periódicamente un snapshot de la base de datos Azure SQL a un servidor local, creando una copia consultable e independiente que permita la generación de reportes, análisis cruzado entre catálogos y alimentación de proyectos de impacto, sin depender del proveedor para el acceso a los datos.

Análisis de escenarios de acceso

La estrategia técnica a implementar depende directamente del nivel de acceso que el proveedor esté dispuesto a otorgar. Se han identificado cuatro escenarios posibles, ordenados del más favorable al menos favorable.

Escenario A: Credenciales SQL de solo lectura

Clasificación: Escenario ideal.

Se solicita al proveedor un usuario SQL con permisos exclusivos de lectura (db_datareader), el endpoint del servidor y la apertura de firewall para la IP del servidor local.

Este escenario habilita el uso de todas las herramientas de exportación disponibles (sqlpackage, bcp, sqlcmd, DuckDB), permite la generación de snapshots completos o selectivos, y ofrece la máxima flexibilidad para automatización.

Modelo de solicitud al proveedor: «Requerimos un usuario SQL de solo lectura (db_datareader) en la base de datos [NOMBRE_DB] alojada en Azure SQL, junto con el endpoint del servidor y la apertura de firewall para nuestra IP [IP_SERVIDOR]. El usuario será utilizado exclusivamente para consultas de lectura y generación de respaldos periódicos de nuestros propios datos.»

Escenario B: Connection string limitada

El proveedor proporciona una cadena de conexión con acceso restringido a vistas específicas o un subconjunto de esquemas, sin acceso a la totalidad de las tablas.

Permite la exportación de las vistas y tablas autorizadas, generando un snapshot parcial pero funcional. Requiere mapear las vistas disponibles y adaptar los scripts de exportación.

Escenario C: Archivos de export del proveedor

El proveedor no otorga acceso directo pero se compromete a entregar periódicamente archivos de exportación (BACPAC o CSV) depositados en un contenedor de Azure Blob Storage o vía carpeta compartida.

Funcional pero parcialmente dependiente: la frescura de los datos queda sujeta a la frecuencia de entrega del proveedor.

Escenario D: Acceso exclusivo vía API

Clasificación: Escenario menos deseable.

No existe acceso SQL. Los datos solo pueden extraerse a través de la API REST de la aplicación del proveedor, endpoint por endpoint, con paginación y límites de tasa.

Es viable pero implica un esfuerzo de implementación significativamente mayor. La prioridad debe ser negociar al menos el Escenario B.

Flujo de datos por ejecución

Cada ejecución del proceso automatizado sigue la siguiente secuencia:

  1. Validar credenciales: lectura desde almacén seguro (Credential Manager), verificación de vigencia.
  2. Verificar conectividad: test TCP al puerto 1433, query de prueba SELECT 1. Si falla, retry con backoff exponencial; si persiste, notificar y abortar.
  3. Exportar datos: según la estrategia definida — BACPAC completo o tablas individuales vía bcp/DuckDB.
  4. Importar a motor local: BACPAC a SQL Server Express, o CSV/Parquet a DuckDB.
  5. Validar integridad: comparar conteos de registros entre origen y destino, verificar tablas no vacías.
  6. Limpiar snapshots antiguos: aplicar política de retención.
  7. Notificar resultado: log local, webhook o email en caso de fallo.

Métodos de exportación

Se dispone de tres métodos para extraer datos de Azure SQL. Cada uno tiene características distintas en cuanto a completitud, velocidad e impacto sobre la base de producción.

Método BACPAC — Snapshot completo

Utiliza sqlpackage /Action:Export para generar un archivo .bacpac que contiene la estructura completa de la base de datos (tablas, vistas, stored procedures, índices, constraints) junto con todos los datos.

Ventajas: copia fiel y completa; restaurable a SQL Server Express local.

Desventajas: alto consumo de DTU en la base de producción; más lento que otros métodos.

Método BCP — Export de tablas individuales

Utiliza la utilidad bcp (Bulk Copy Program) para exportar tablas o vistas individuales como archivos CSV. Permite seleccionar exactamente qué datos exportar mediante queries personalizadas.

Ventajas: bajo consumo de DTU; rápido; exporta solo lo necesario; queries filtrables.

Desventajas: no incluye estructura de la base; no es transaccionalmente consistente entre tablas; requiere generar headers por separado.

Recomendado para: generación de reportes entre catálogos específicos donde no se requiere la base de datos completa.

Método DuckDB directo a Parquet

Utiliza la extensión comunitaria mssql de DuckDB para conectarse directamente a Azure SQL y exportar tablas a formato Apache Parquet con compresión ZSTD.

Ventajas: un solo paso (Azure SQL a Parquet); formato columnar optimizado para analytics; archivos inmediatamente consultables con DuckDB.

Desventajas: extensión experimental; soporte limitado de tipos de datos.

Si la extensión mssql de DuckDB presenta inestabilidad, se utiliza como fallback el pipeline: bcp a CSV, seguido de importación a DuckDB.

Recomendación: DuckDB es el motor óptimo para el caso de uso de reportes entre catálogos, dado que:

  • No requiere instalación de servicio: se abre como un archivo.
  • Motor columnar optimizado para aggregaciones y JOINs analíticos.
  • Lee archivos Parquet directamente sin necesidad de importarlos.
  • Integración nativa con Python (import duckdb) para scripts de reportes.

Si se requiere compatibilidad T-SQL exacta con la base de origen (por ejemplo, para reutilizar stored procedures existentes), SQL Server Express es la alternativa adecuada.

Seguridad y manejo de credenciales

Principios de seguridad

  1. Las credenciales nunca se almacenan en texto plano dentro de scripts o archivos de configuración.
  2. Las credenciales nunca se incluyen en repositorios de código.
  3. Se utilizan mecanismos de almacenamiento encriptado nativos del sistema operativo.
  4. Se establece un calendario de rotación de credenciales con el proveedor.

Automatización y política de retención

Mecanismo de ejecución

El script de exportación se registra como tarea programada en Windows Task Scheduler, configurado para ejecutarse cada hora (o según la frecuencia que se defina). La tarea incluye:

  • Ejecución como usuario SYSTEM con privilegios elevados.
  • Reintento automático (2 reintentos, intervalo de 5 minutos).
  • Límite de ejecución de 2 horas.
  • Ejecución solo si hay red disponible.
  • Prevención de instancias concurrentes.

Validación y criterios de aceptación

Al completar cada ciclo de exportación e importación, el script ejecuta automáticamente las siguientes validaciones:

  1. Comparación de conteos: se verifica que el número de registros de cada tabla en el snapshot local coincida con el conteo en Azure SQL. Tolerancia: 0% para catálogos estáticos, <1% para tablas transaccionales en movimiento.
  2. Verificación de tablas no vacías: ninguna tabla definida en el esquema de export debe quedar sin registros.
  3. Prueba de consulta cruzada: se ejecuta un JOIN entre al menos dos tablas del snapshot local para confirmar la integridad relacional.

Criterios de aceptación del sistema completo

  • Todas las tablas definidas se exportaron sin error.
  • El conteo de registros coincide entre Azure y el snapshot local.
  • Se puede ejecutar un JOIN entre 2+ tablas del snapshot local.
  • El proceso completo termina dentro de la ventana de tiempo definida.
  • Los snapshots antiguos se eliminan según la política de retención.
  • En caso de fallo, se genera notificación y el snapshot anterior permanece disponible.
  • Se genera exitosamente un reporte real desde el snapshot local.

Plan de implementación por fases

Fase 0: Negociación con proveedor

Prerequisito. Se envía solicitud formal al proveedor para obtener acceso de lectura (Escenario A). Si se niega, se negocia por orden de preferencia: Escenario B, luego C. Se documenta el acuerdo alcanzado.

Entregable: credenciales validadas o mecanismo de acceso acordado.

Fase 1: Setup de infraestructura local

Instalación de herramientas CLI, configuración de credenciales en Credential Manager y validación de conectividad TCP al servidor Azure SQL.

Entregable: conexión funcional verificada con query SELECT @@VERSION.

Fase 2: Diseño y primer export manual

Catalogar tablas disponibles, estimar tamaños, decidir estrategia de export (BACPAC completo vs. tablas selectas) y ejecutar el primer snapshot manualmente para validar el flujo.

Entregable: primer snapshot local funcional; matriz de tablas documentada.

Fase 3: Pipeline automatizado

Desarrollo del script PowerShell completo con logging, retry y notificaciones. Configuración del import a motor local e inscripción en Task Scheduler. Periodo de prueba monitoreada de 24–48 horas.

Entregable: pipeline automatizado operando.

Fase 4: Validación y hardening

Implementación de validación de integridad post-import, configuración de política de retención, pruebas de escenarios de fallo y generación del primer reporte real desde snapshot local.

Entregable: sistema validado, documentado y primer reporte de impacto generado.

Consideraciones legales, técnicas y de continuidad

Consideraciones legales y contractuales

  • Propiedad de datos: los datos generados en la operación del negocio son propiedad de la organización, no del proveedor. Esto debe estar explícito en el contrato.
  • Obligación de acceso: el proveedor tiene obligación contractual (o debería tenerla) de facilitar acceso a los datos propios del cliente.
  • Portabilidad: si el contrato actual no incluye cláusula de portabilidad de datos, se recomienda negociar su inclusión en la próxima renovación.
  • Cumplimiento normativo: verificar que la copia local de datos cumpla con regulaciones aplicables de protección de datos personales.

Consideraciones técnicas

  • Impacto en producción: la exportación consume DTU de la base en Azure. Acordar con el proveedor ventanas de export o habilitación de read replica.
  • Ancho de banda: un export de 5 GB sobre internet puede tardar 10–30 minutos. Programar en horarios de bajo tráfico.
  • Consistencia transaccional: el export tabla por tabla vía bcp no es transaccionalmente consistente entre tablas. Para reportes periódicos esto es aceptable; para consistencia estricta, utilizar BACPAC.
  • Cambios de esquema: si el proveedor modifica la estructura de tablas sin previo aviso, el script de export puede fallar. Implementar detección de cambios de schema.

Consideraciones de continuidad

  • Revocación de acceso: los snapshots locales existentes permanecen disponibles como respaldo de último recurso.
  • Fallo del servidor local: configurar los snapshots en un segundo disco o ruta de respaldo.
  • Cambio de IP: solicitar actualización de regla de firewall al proveedor, o utilizar VPN con IP fija.
